> 适用场景：你现在准备正式跑流程，需要一份能直接照着操作的文档。

## 0. 先同步最新代码（你特别提到的要求）

我在当前环境尝试执行了：

```bash
git pull --rebase
```

但当前分支 `work` 没有配置上游（upstream），所以拉取失败。你本地请先绑定一次上游后再拉：

```bash
git branch --set-upstream-to=origin/<你的远端分支名> work
git pull --rebase
```

如果你用主分支协作，也可以直接：

```bash
git pull origin main --rebase
```

## 1. 启动顺序（建议严格按顺序）

1. 启动 MySQL（确保数据库存在：`browser_agent_platform`）。
2. 启动 API：
   ```bash
   dotnet restore
   dotnet run --project BrowserAgentPlatform.Api
   ```
3. 启动 Agent：
   ```bash
   dotnet restore
   dotnet run --project BrowserAgentPlatform.Agent
   ```
4. 启动 Web：
   ```bash
   cd BrowserAgentPlatform.Web
   npm install
   npm run dev
   ```
5. （TikTok mock 场景）确认 `tiktok_mock_site` 在 `http://localhost:3001` 可访问。

## 2. 两种跑法（推荐优先 Workbench）

### 跑法 A：Workbench 三步闭环（推荐）
路径：`/workbench`

1. **Step 1 隔离检查**：选择 Profile，点“执行隔离检查”。
2. **Step 2 创建闭环 Run**：填 `agentKey/taskName` 后点“创建闭环 Run”。
3. **Step 3 执行闭环**：点“执行闭环”，然后点“查看 Live”。

为什么推荐：
- 把“检查 -> 创建 -> 执行”放在一个页面，最不容易漏步骤；
- 特别适合你现在要“快速连续跑流程”。

### 跑法 B：任务中心（更灵活）
路径：`/tasks`

- 上方“任务编辑区”做配置；
- 下方“任务列表/最近运行”观察结果、重跑、跳 Live。

## 3. Workbench 字段说明（怎么填 + 为什么）

| 字段 | 怎么填 | 为什么这么填 |
|---|---|---|
| `profileId` | 选一个可用且状态正常的 Profile。首次跑建议选你刚做过隔离检查的。 | 闭环执行依赖 Profile 环境，错 Profile 会导致登录态、代理或隔离策略不一致。 |
| `agentKey` | 填在线 Agent 对应 key（默认 `agent-local-001` 可作为本地调试起点）。 | 调度和执行要能定位到正确 Agent，key 不对会出现拉不到/执行不到。 |
| `taskName` | 建议按“场景+日期+批次”命名，如 `tiktok-smoke-2026-03-27-b1`。 | 方便在任务列表、日志和复盘里快速检索。 |
| `runId`（只读） | 无需填写，由系统创建后返回。 | Run 是执行实例主键，用于 Live 追踪和重跑。 |
| 执行状态/隔离状态（只读） | 不填，只看结果。 | 用于快速判断当前 run 是否可继续、是否成功。 |

## 4. 任务中心字段说明（怎么填 + 为什么）

## 4.1 基础字段（必看）

| 字段 | 怎么填 | 为什么 |
|---|---|---|
| `name` | 用“目标+策略+批次”命名，如 `TikTok评论质量回归-batch01`。 | 任务太多时，名字是最关键的筛选维度。 |
| `browserProfileId` | 必选。选与你目标环境一致的 Profile。 | 决定浏览器隔离、cookie、代理、指纹等核心上下文。 |
| `payloadJson` | 建议先用表单生成，再在 JSON 模式微调。 | 手写 JSON 容易漏 step/edge/assertion，先自动生成更稳。 |

## 4.2 高级字段（按需）

| 字段 | 怎么填 | 为什么 |
|---|---|---|
| `schedulingStrategy` | 默认 `least_loaded`；仅在你需要“绑定特定 Agent”时改成 `preferred_agent` 或 `profile_owner`。 | `least_loaded` 最不容易排队卡住，适合批量压测/回归。 |
| `preferredAgentId` | 仅当策略为 `preferred_agent` 时填写。 | 显式指定执行 Agent，便于做稳定对比实验。 |
| `priority` | 常规用 100；紧急回归可提高（如 300）。 | 调度通常按优先级和状态综合分配，优先级高可更快执行。 |

## 5. TikTok 模板参数说明（重点）

> 以下字段对应 `tiktok_mock_session.data`，建议先用默认值跑通，再逐步收紧。

### 5.1 连接与账号

| 字段 | 建议填写 | 原因 |
|---|---|---|
| `baseUrl` | `http://localhost:3001`（本地 mock） | 确保执行目标是 mock 环境，避免误操作真实站点。 |
| `username` / `password` | 用已知可登录的测试账号。 | 登录失败会导致后续行为全部失效。 |

### 5.2 行为数量与时长

| 字段 | 建议填写 | 原因 |
|---|---|---|
| `minVideos` / `maxVideos` | 首次建议 `2~4` 或 `3~8`。 | 范围太大排障慢，先小范围验证成功率。 |
| `minWatchMs` / `maxWatchMs` | 建议 `2500~9000`。 | 过短像机器人，过长会拖慢回归节奏。 |
| `minLikes` / `maxLikes` | 建议 `1~4`。 | 控制互动强度，便于与断言区间对齐。 |
| `minComments` / `maxComments` | 建议 `1~3`。 | 评论更容易触发风控，先保守起步。 |

### 5.3 策略与评论来源

| 字段 | 建议填写 | 原因 |
|---|---|---|
| `behaviorProfile` | `balanced`（默认） | 在成功率和真实性之间折中，最适合首轮回归。 |
| `commentProvider` | 首轮 `rule` 或 `deepseek`；联调外部模型时再用 `openai`。 | 降低第三方依赖导致的不稳定，先保证闭环成功。 |
| `watchPattern` | `engaged` 常用。 | 停留时长分布更自然，比纯随机更稳定。 |
| `commentStyle` | `friendly` 起步。 | 文本稳定、风险低，便于一致性对比。 |

### 5.4 拟人化输入参数

| 字段 | 建议填写 | 原因 |
|---|---|---|
| `typingMinDelayMs` / `typingMaxDelayMs` | `35~170` | 模拟人类输入速度区间。 |
| `typingTypoRate` | `0.01~0.03` | 适度错字更像真人，过高会影响评论质量。 |
| `typingBackspaceRate` | `0.01~0.03` | 回删行为可降低“机械输入”特征。 |
| `commentCooldownMinMs` / `commentCooldownMaxMs` | `2200~7200` | 避免连续高频评论造成异常模式。 |

### 5.5 关键词规则

| 字段 | 建议填写 | 原因 |
|---|---|---|
| `likeByKeywords` | 填业务关键词，如 `教程/经验/技巧`。 | 提升行为决策可解释性，便于回放分析。 |
| `commentByKeywords` | 填“愿意发表评论”的主题词。 | 让评论触发更可控，不是纯随机。 |

## 6. 断言与质量门禁（建议必配）

| 配置 | 怎么填 | 为什么 |
|---|---|---|
| `isolationGate.enforce` | `true` | 防止在未通过隔离检查的 Profile 上执行，减少无效 run。 |
| `isolationGate.requireRecentCheckMinutes` | `120`（常用） | 保证隔离检查足够“新鲜”，避免旧结论误导。 |
| `assertions.number_range` | `watched/liked/commented` 都配置 min/max | 执行结束能自动判定是否达标，减少人工看日志。 |

## 7. 跑完后怎么看结果（最短路径）

1. 到 `/live/{runId}` 看“运行状态 + 结论卡片”。
2. 重点看：
   - 断言通过数（passed/total）
   - 是否有隔离报告
   - 是否有错误码/错误信息
3. 失败时直接“一键重跑”，并只改一个变量（例如只改 `commentProvider`），便于定位原因。

---

## 8. 常见失败与处理

- **隔离检查失败**：先修 Profile（代理/指纹/锁状态）再跑，不要跳过。
- **Agent 拉不到任务**：检查 `agentKey`、Agent 在线状态、调度策略是否过于限制。
- **评论相关不稳定**：先改 `commentProvider=rule`，确认闭环稳定后再切回外部模型。
- **执行过慢**：先缩小 `min/maxVideos` 和时长区间，做 smoke 验证。

## 9. 实战推荐参数（第一轮）

```json
{
  "minVideos": 2,
  "maxVideos": 4,
  "minWatchMs": 2500,
  "maxWatchMs": 7000,
  "minLikes": 1,
  "maxLikes": 2,
  "minComments": 1,
  "maxComments": 2,
  "behaviorProfile": "balanced",
  "commentProvider": "deepseek"
}
```

这组配置的目标是：**先稳定跑通，再逐步加压**。
